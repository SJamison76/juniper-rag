Juniper Config Critique
File  : config.txt
Focus : harden this config
============================================================

SUMMARY
The device has a reasonable logging baseline but contains several significant hardening gaps including an enabled cleartext management service, world-readable sensitive log files, a shared TACACS account with full access, and no loopback firewall filter protecting the RE.

ISSUES

ISSUE 1: XNM cleartext (JUNOScript over TCP 3221) enabled
delete system services xnm-clear-text
Cleartext management protocol exposes credentials and session data on the wire.

ISSUE 2: No SSH hardening on management services
set system services ssh root-login deny
set system services ssh max-sessions-per-connection 1
set system services ssh connection-limit 5
set system services ssh rate-limit 5
Root can log in directly over SSH; connection limits prevent brute-force attacks.

ISSUE 3: auth.log is world-readable
delete system syslog file auth.log archive world-readable
Authorization logs contain sensitive authentication data and must not be globally readable.

ISSUE 4: login-attempts log has no archive size or world-readable set consistently but more critically the match regex has spaces around the pipe which will break matching
set system syslog file login-attempts match "(login_failed|login_pam_authentication_error|login_pam_error|login_timed_out|sshd_login_failed)"
Spaces inside the regex around the pipe operator cause the pattern to fail to match intended strings.

ISSUE 5: security.log is world-readable and contains password-related matches
delete system syslog file security.log archive world-readable
Security and password-related log data must not be readable by all local users.

ISSUE 6: Remote user account with FULL-ACCESS and no authentication configured
delete system login user remote class FULL-ACCESS
set system login user remote class READ-ACCESS
A shared account with full superuser access and no per-user authentication is a critical accountability and privilege gap.

ISSUE 7: No loopback firewall filter protecting the Routing Engine
set firewall family inet filter protect-re term allow-ssh from source-address <mgmt-prefix>/32
set firewall family inet filter protect-re term allow-ssh from protocol tcp
set firewall family inet filter protect-re term allow-ssh from destination-port 22
set firewall family inet filter protect-re term allow-ssh then accept
set firewall family inet filter protect-re term allow-netconf from source-address <mgmt-prefix>/32
set firewall family inet filter protect-re term allow-netconf from protocol tcp
set firewall family inet filter protect-re term allow-netconf from destination-port 830
set firewall family inet filter protect-re term allow-netconf then accept
set firewall family inet filter protect-re term deny-all then discard
set interfaces lo0 unit 0 family inet filter input protect-re
Without a loopback filter any host can attempt to reach management services directly on the RE.

ISSUE 8: me0 uses DHCP for management addressing
set interfaces me0 unit 0 family inet address <static-mgmt-ip>/prefix
A management interface with a dynamic address is unreliable and cannot be consistently filtered or audited.

ISSUE 9: No login announcement or banner configured
set system login message "UNAUTHORIZED ACCESS IS PROHIBITED. This system is monitored."
A pre-login banner is required for legal enforceability of access policies.

ISSUE 10: No syslog remote host configured
set system syslog host <syslog-server-ip> any any
Local-only logging means logs are lost if the device is compromised or fails.

ISSUE 11: READ-ACCESS class has no deny-configuration stanza
set system login class READ-ACCESS deny-configuration "(groups)|(system)|(interfaces)|(routing-instances)"
Without deny-configuration, view permissions alone do not prevent configuration reads of sensitive stanzas via NETCONF.

ISSUE 12: No idle-timeout on root account and root has a password but no class restrictions
set system login class FULL-ACCESS idle-timeout 10
Root sessions do not inherit login class idle-timeout; 30 minutes is excessive for privileged access.

RECOMMENDATIONS

REC 1: Protect the loopback filter configuration once applied
protect interfaces lo0 unit 0 family inet filter
Prevents accidental or unauthorized removal of the RE protection filter.

REC 2: Add NETCONF access control via SSH subsystem restriction to known management hosts only
set system services netconf ssh connection-limit 5
Limits concurrent NETCONF sessions to reduce exposure of the automation interface.

REC 3: Enable syslog structured-data for easier SIEM ingestion
set system syslog file messages structured-data
Structured syslog output improves parsing reliability in external log management systems.

REC 4: Set archive world-readable off explicitly on all remaining log files
delete system syslog file bpdu-events.log archive world-readable
delete system syslog file daemon.log archive world-readable
delete system syslog file firewall.log archive world-readable
delete system syslog file health.log archive world-readable
delete system syslog file interfaces.log archive world-readable
delete system syslog file messages archive world-readable
Least-privilege principle applies to log file permissions even for non-sensitive logs.

REC 5: Add NTP authentication to ensure time integrity for log correlation
set system ntp authentication-key 1 type md5 value <key>
set system ntp server <ntp-server-ip> key 1
Unauthenticated NTP allows time manipulation which undermines log forensics.

CORRECT
Login classes have idle-timeout 30 and login-alarms configured.
NETCONF is configured with both rfc-compliant and yang-compliant flags.
Comprehensive syslog coverage across auth, firewall, daemon, config-change, and interactive-commands.
READ-ACCESS class correctly uses deny-commands to block ssh and telnet lateral movement.
System timezone is explicitly set.
Syslog timestamps include year and millisecond precision for accurate log correlation.
config.log captures change-log at info level for configuration audit trail.
